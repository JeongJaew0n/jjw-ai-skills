# 결정 축 카탈로그

이 파일은 2단계에서 이번 작업의 결정 축을 선별할 때 참고한다. 모든 축을 매번 묻지 않는다. 요구사항과 코드베이스를 보고, 이번 작업에서 실제로 결정이 갈리는 축만 고른다.

각 축은 "스펙에서 확정해야 하는 질문"과 "어떤 작업에서 중요해지는가"로 정리한다.

## Atomicity
**확정 질문:** 이 동작과 부수 효과가 같은 트랜잭션으로 묶여야 하는가, 아니면 분리되어야 하는가.
**중요해지는 작업:** 상태 변경과 외부 호출이 한 흐름에 섞이는 경우, 한쪽 실패가 다른 쪽을 되돌려야 하는지 갈리는 경우.
**갈리는 지점:** 트랜잭션 내부에서 직접 호출 vs commit 이후 outbox 발행 vs 별도 작업 큐.

## Idempotency
**확정 질문:** 같은 이벤트나 요청이 두 번 처리되어도 결과가 한 번만 반영되어야 하는가.
**중요해지는 작업:** 이벤트 기반 처리, 재시도가 있는 흐름, 메시지 큐 소비, 외부 webhook 수신.
**갈리는 지점:** 중복 방지 키 기준, 멱등 저장소 사용 여부, at-least-once 와 exactly-once 중 무엇을 보장하는가.

## Failure-mode
**확정 질문:** 부수 효과가 실패하면 본 동작 상태에 영향을 주어야 하는가, 어떻게 복구되는가.
**중요해지는 작업:** 외부 시스템 호출이 끼는 경우, 부분 실패가 가능한 모든 다단계 흐름.
**갈리는 지점:** 실패 시 rollback vs 본 동작은 성공으로 두고 부수 효과만 재시도, 재시도 횟수와 백오프, 최종 실패 처리.

## Observability
**확정 질문:** 이 동작의 요청과 실패는 어떤 로그, 이벤트, 메트릭으로 추적되어야 하는가.
**중요해지는 작업:** 비동기 흐름, 재시도 흐름, 운영 중 디버깅이 어려울 흐름.
**갈리는 지점:** 어떤 지점에 로그를 남기는가, 실패를 메트릭으로 노출하는가, 추적 ID 를 전파하는가.

## Side-effects
**확정 질문:** 외부 시스템 호출이 언제, 어떤 방식으로 발생해야 하는가.
**중요해지는 작업:** 메일, 알림, 결제, 외부 API 연동처럼 시스템 밖으로 나가는 호출이 있는 경우.
**갈리는 지점:** 동기 호출 vs 비동기 분리, 호출 시점이 commit 전인가 후인가.

## Compatibility
**확정 질문:** 기존 API 계약, 이벤트 스키마, 저장 포맷이 바뀌는가, 바뀐다면 하위 호환을 지키는가.
**중요해지는 작업:** 기존 엔드포인트나 이벤트를 건드리는 변경, 공유 스키마 변경, 클라이언트가 여럿인 API.
**갈리는 지점:** 필드 추가만 vs 변경/삭제, 버저닝 전략, 마이그레이션 순서.

## Security / Authorization
**확정 질문:** 누가 이 동작을 호출할 수 있는가, 권한 검사는 어느 계층에서 일어나는가.
**중요해지는 작업:** 사용자 데이터 접근, 권한별로 동작이 갈리는 기능, 관리자/일반 사용자 구분.
**갈리는 지점:** 권한 검사 위치, 권한 없는 접근의 응답, 민감 데이터 노출 범위.

## Tenant Isolation
**확정 질문:** 멀티테넌트 환경에서 데이터와 동작이 테넌트 경계를 넘지 않는가.
**중요해지는 작업:** 테넌트별로 격리되어야 하는 데이터 조회, 생성, 집계.
**갈리는 지점:** 테넌트 필터가 모든 쿼리에 강제되는가, 공유 리소스에서의 격리 방식.

## Concurrency
**확정 질문:** 동시 요청이나 동시 처리가 안전한가, 경합 시 어떤 동작이 보장되는가.
**중요해지는 작업:** 같은 자원을 여러 요청이 동시에 건드리는 경우, 재고/잔액처럼 경합이 결과를 바꾸는 경우.
**갈리는 지점:** 락 전략, 낙관적 vs 비관적 동시성, 경합 패배 시 처리.

## 데이터 모델 / 마이그레이션
**확정 질문:** 스키마 변경이 필요한가, 기존 데이터는 어떻게 이전되는가.
**중요해지는 작업:** 새 필드/테이블 추가, 저장 구조 변경, 무중단 배포가 필요한 변경.
**갈리는 지점:** 마이그레이션 순서, 기존 데이터 백필 방식, 배포와 스키마 변경의 선후 관계.

---

## 선별 가이드

축을 고를 때 다음을 자문한다:

- 이 요구사항에서 **외부로 나가는 호출**이 있는가 → Side-effects, Failure-mode, Atomicity
- **이벤트나 재시도**가 끼는가 → Idempotency, Observability
- **기존 것을 건드리는가** → Compatibility, 데이터 모델/마이그레이션
- **누가 호출하는지가 갈리는가** → Security/Authorization, Tenant Isolation
- **같은 자원에 동시 접근**이 있는가 → Concurrency

해당하지 않는 축은 생략하되, 생략한 이유를 사용자에게 한 줄로 밝힌다. 빠진 결정이 의도된 것인지 사용자가 판단할 수 있어야 한다.
