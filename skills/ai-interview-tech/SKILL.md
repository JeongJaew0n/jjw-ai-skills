# AI Interview Tech

## 활성화 조건

사용자가 `/ai-interview-tech`, `$ai-interview-tech` 또는 `ai-interview-tech`를 직접 언급해 호출한 경우에만 이 스킬을 실행한다. 일반적인 구현·설계 요청이나 기술 결정이 필요한 상황만으로 이 스킬을 자동 선택하지 않는다.

## 이 스킬이 푸는 문제

AI가 코드의 상당 부분을 작성하면 사람의 통제 지점이 코드 작성에서 의도 전달과 검증으로 옮겨간다. 그런데 직접 코드를 쓰지 않은 만큼, 결과물이 어떤 판단을 거쳐 만들어졌는지 다시 확인하는 인지 부하가 생긴다.

문제의 뿌리는 리뷰가 아니라 그 앞에 있다. 비즈니스 요구사항은 자연어로 자연스럽게 들리지만, 코드 수준에서 보면 그 안에 여러 기술 결정이 숨어 있다. 자연어 요구사항을 그대로 구현 프롬프트로 넘기면 그 결정이 전부 AI의 해석으로 넘어가고, 사람이 함께 논의하지 않은 결정으로 구현 어딘가에 박힌다.

이 스킬의 목적은 단 하나다. **요구사항 뒤에 숨은 기술 결정을 구현 전에 드러내고, 사람의 판단이 필요한 부분을 숨기지 않고 함께 확정한다.**

핵심은 요구사항을 길게 쓰는 것이 아니다. 어떤 결정이 필요한지 드러나는 구조를 만드는 것이다.

## 무엇을 하지 말아야 하는가

- 요구사항을 받자마자 구현으로 직행하지 않는다. 구현은 이 인터뷰가 끝난 뒤다.
- 결정을 사용자에게 통째로 떠넘기지 않는다. "어떻게 할까요?"라고 넓게 묻는 것은 인지 부하를 사용자에게 전가하는 것이다.
- 추상적인 질문을 던지지 않는다. "실패 시 어떻게 처리할까요"는 사용자가 코드베이스 구조를 다시 머릿속에 올리게 만든다.
- 결정을 조용히 내려버리지 않는다. AI가 임의로 정한 결정은 리뷰에서 역추적 비용을 만든다.

## 워크플로우

### 1단계 — 코드베이스를 먼저 읽는다

질문하기 전에 현재 코드베이스를 읽는다. 같은 "알림을 보낸다"는 요구사항이라도 프로젝트마다 적절한 구현이 다르다. 어떤 코드베이스에는 이미 outbox 패턴이 있고, 어떤 코드베이스는 동기 외부 호출을 기존 컨벤션으로 쓴다. 어떤 서비스는 이벤트 발행을 표준으로 삼고, 어떤 서비스는 작업 큐나 배치 재시도를 쓴다.

읽으면서 확인할 것:
- 이번 작업과 맞닿는 기존 패턴과 컨벤션은 무엇인가
- 트랜잭션 경계, 에러 처리, 재시도, 이벤트 발행이 기존에 어떻게 되어 있는가
- 비슷한 기능이 이미 있다면 그 구현이 어떤 결정을 따랐는가

코드베이스를 읽지 않고 던지는 질문은 추상적일 수밖에 없다. 결정이 갈리는 지점은 코드 안에 있다.

### 2단계 — 이번 작업의 결정 축을 선별한다

요구사항과 코드베이스를 본 뒤, 이번 작업에서 실제로 결정이 갈리는 축만 골라낸다. 모든 축을 매번 묻지 않는다. 어떤 변경에서는 Atomicity와 Failure-mode가 핵심이고, 어떤 변경에서는 Compatibility와 Security가 핵심이 된다.

결정 축 카탈로그는 `references/decision-axes.md`에 있다. 처음 작업 유형을 만나거나 어떤 축이 중요한지 판단이 서지 않을 때 그 파일을 읽는다.

선별이 끝나면 사용자에게 먼저 지도를 보여준다. 무엇을 논의하고, 무엇을 왜 생략하는지 한눈에 보이게 한다. 예:

```
이번 작업은 알림 발송 기능 추가로 보입니다.
핵심 결정은 Atomicity / Idempotency / Failure-mode 입니다.
Observability 는 실패 추적을 위해 짧게 확인하고,
Compatibility 는 기존 API 계약 변경이 없으므로 생략하겠습니다.
```

생략하는 축은 생략 이유를 밝힌다. 그래야 사용자가 빠진 결정이 의도된 것인지 판단할 수 있다.

### 3단계 — grill-me 방식으로 한 축씩 인터뷰한다

선별한 축을 결정이 갈리는 순서대로 하나씩 진행한다. 각 축마다 네 가지 규칙을 지킨다.

1. **한 번에 하나의 질문만 한다.** 여러 결정을 한꺼번에 묻지 않는다.
2. **질문에 추천 답을 함께 제시한다.** AI가 코드베이스를 근거로 선택지를 좁히고 추천안을 낸다. 사용자는 추천을 보고 확정하거나 뒤집는다.
3. **추상 원칙보다 관련 코드를 먼저 보여준다.** 선택지를 코드로 보여줘야 질문이 구체화된다.
4. **결정이 갈리는 순서대로 묻는다.** 앞 결정이 뒤 결정을 좌우하는 경우, 앞엣것을 먼저 확정한다.

추상적인 질문과 코드 기반 질문의 차이:

**나쁜 질문:**
```
알림 실패 시 어떻게 처리할까요?
```
너무 넓다. 사용자가 맥락과 코드베이스 구조를 다시 떠올려야 한다.

**좋은 질문:**
```kotlin
// option A — 트랜잭션 안에서 바로 알림 전송
@Transactional
fun completeOrder(orderId: OrderId) {
    val order = orderRepo.get(orderId)
    order.markCompleted()
    notificationClient.send(order)
}

// option B — commit 후 발행할 outbox만 저장
@Transactional
fun completeOrder(orderId: OrderId) {
    val order = orderRepo.get(orderId)
    order.markCompleted()
    outboxRepo.save(NotificationRequested(order.id))
}
```
```
이 코드베이스에는 이미 outbox 테이블과 발행 worker 가 있습니다.
알림 실패가 주문 완료를 rollback 시키면 안 된다면,
B 처럼 commit 이후 발행하는 방식이 더 적합합니다.
이 방향으로 확정할까요?
```

인터뷰의 목적은 판단을 떠넘기는 것이 아니라 반대다. AI가 코드베이스를 먼저 읽고, 가능한 선택지를 좁히고, 추천안을 제시한다. 사용자는 그 지점에서 "이 방향이 맞다" 또는 "아니다"만 결정하면 된다. 그래서 인지 부하가 줄어든다.

### 4단계 — 확정된 결정만 스펙에 기록한다

각 축이 확정될 때마다 스펙에 반영한다. 아직 확정되지 않은 것, 사용자가 보류한 것은 기록하지 않는다. 잡담으로 흐른 부분도 기록하지 않는다.

인터뷰가 끝나면 아래 형태로 결정 스펙을 정리한다. 이 스펙은 나중에 리뷰를 "전체 코드 재해석"이 아니라 "합의한 결정이 코드에 반영됐는지 확인"하는 일로 바꾸는 산출물이다.

```markdown
# 결정 스펙: <기능 이름>

## 요구사항
<사용자가 준 비즈니스 요구사항 한 줄>

## 확정된 결정
| 결정 축 | 확정 내용 | 근거 |
|---|---|---|
| Atomicity | 주문 완료 commit 이후 outbox 로 알림 분리 | 알림 실패가 주문 완료를 rollback 시키면 안 됨, 기존 outbox worker 존재 |
| Idempotency | 주문 ID 기준 중복 방지 키 사용 | 이벤트 재처리 시 알림 1회 보장 |
| Failure-mode | 알림 실패는 주문 완료에 영향 없음, worker 가 재시도 | ... |

## 논의에서 생략한 축
| 결정 축 | 생략 이유 |
|---|---|
| Compatibility | 기존 주문 완료 API 계약 변경 없음 |

## 리뷰 검증 체크리스트
- [ ] 스펙에서 확정한 트랜잭션 경계가 코드에 반영됐는가
- [ ] 중복 방지 키가 합의한 기준대로 사용됐는가
- [ ] 실패한 알림이 주문 완료 상태를 흔들지 않고 재시도 가능하게 남는가
- [ ] 로그와 이벤트가 합의한 관측 지점을 제공하는가
```

## 인터뷰가 끝난 뒤

스펙이 확정되면 그때 구현으로 넘어간다. 구현 프롬프트에는 자연어 요구사항이 아니라 확정된 결정 스펙을 근거로 넘긴다.

리뷰 단계에서는 위의 검증 체크리스트를 기준으로 본다. 코드가 길어도 모든 줄을 같은 밀도로 읽지 않는다. 스펙에서 합의한 결정이 실제 구현에 어떻게 반영됐는지를 먼저 본다.

### 결정 스펙을 영속화한다

확정된 결정 스펙과 검증 체크리스트는 **리뷰 시점·다음 세션·다른 PC에서 다시 필요**하다. 세션 메모리에만 두면 정작 쓸 때 사라진다. 그래서 다음 기준으로 파일에 박제한다.

- **결정 축이 2개 이상이고 구현이 여러 세션에 걸칠 작업** → 영속화한다.
- **축 1개짜리 단발 변경** → 대화 메모리로 충분하므로 생략한다. (잡동사니 폴더를 만들지 않는다.)

영속화 방법은 우선순위가 있다.

1. **`ai-plan-memory` 스킬이 있으면 그쪽에 위임한다.** 새 저장 규약을 따로 만들지 않는다. 인터뷰 산출물은 그 스킬의 세 파일과 자연스럽게 대응한다.
   - 확정된 결정 표 + 근거 → `spec.md` / `context.md`
   - 논의에서 생략한 축 → `context.md`
   - 리뷰 검증 체크리스트 → `checklist.md`
   - 호출 시 위에서 정리한 결정 스펙을 그대로 넘겨 채워지게 한다.

2. **`ai-plan-memory` 스킬이 없으면, 결정 스펙을 md 파일로 직접 저장한다.** `docs/plans/<slug>/decision-spec.md` 에 4단계에서 정리한 결정 스펙 블록(요구사항 / 확정된 결정 / 생략한 축 / 리뷰 검증 체크리스트)을 그대로 쓴다. `<slug>` 는 기능 이름을 영문 kebab-case 로 정한다.

어느 경로든, 저장한 파일 경로를 사용자에게 짧게 알려 다음 세션이 픽업할 수 있게 한다.

## 전략과 전술의 구분

- **전술**은 grill-me 의 인터뷰 규칙을 빌린다. 한 번에 하나, 추천안 포함, 코드 먼저, 갈리는 순서대로.
- **전략**은 그 질문들을 기술 결정 축으로 제한한다. 인터뷰가 잡담으로 흐르지 않게, 구현 전에 놓치기 쉬운 결정만 다룬다.

이 둘을 함께 지킬 때만 인터뷰가 효과를 낸다. 규칙만 있고 결정 축이 없으면 잡담이 되고, 축만 있고 규칙이 없으면 사용자에게 결정을 떠넘기게 된다.
