# /ai-plan-memory — 작업 계획을 docs/plans/ 에 영속화

## 무엇을 하는 스킬인가

지금까지 대화에서 합의된 "다음에 할 작업"을 **프로젝트 루트의 `docs/plans/<slug>/`** 폴더에
세 파일로 박제한다. 다음 세션에서 이 폴더만 읽으면 무엇을, 왜, 어디까지 했는지 즉시 복원
가능하게 만드는 것이 목적이다.

세 파일의 책임은 엄격하게 분리한다.

| 파일 | 책임 | 누가 갱신하나 |
|---|---|---|
| **spec.md** | 무엇을 만들 것인가. 구현 스펙·범위·완료 조건. | 작업 시작 전 한 번 작성, 스펙 변경 시 갱신 |
| **context.md** | 왜 이 작업이 시작됐는가. 사용자의 원래 요청·결정 이유·기각된 대안. | 시작 시 작성 후 거의 동결 (배경 기록) |
| **checklist.md** | 어디까지 했는가. 단계별 체크박스. | 작업 진행하며 AI 가 계속 갱신 |

## 실행 절차

다음을 순서대로 수행한다.

### 1. 프로젝트 루트 확인

```bash
git rev-parse --show-toplevel 2>/dev/null || pwd
```

`docs/plans/` 경로는 항상 이 루트 기준이다. 서브 디렉터리에서 호출돼도 루트로 정규화한다.

### 2. slug 결정

`docs/plans/<slug>/` 의 `<slug>` 는 영문 kebab-case 권장 (예: `chat-panel-ai-bridge`,
`weight-zero-rule`, `auto-save-server-ssot`).

slug 후보를 정할 때:

- 대화 맥락에서 "이번에 합의된 작업"의 핵심 명사구를 뽑는다
- 한국어 주제는 영문으로 의역 (예: "발생 가중치 0 규칙" → `weight-zero-rule`)
- 너무 일반적이지 않게 (`refactor`, `fix` 단독 금지)

후보가 1개로 분명하면 그대로 진행, 애매하거나 2개 이상 떠오르면 `AskUserQuestion` 으로
사용자에게 1개 선택받는다. **사용자가 명시한 slug 가 있으면 그대로 사용한다.**

### 3. 충돌 확인

```bash
ls docs/plans/<slug>/ 2>/dev/null
```

- 폴더가 없으면 → 새로 만들고 3개 파일 생성
- 폴더가 있으면 → 사용자에게 묻기:
  - 이어서 갱신 (기존 파일 Edit)
  - 새 slug 로 생성 (`<slug>-v2` 등 제안)
  - 덮어쓰기 (위험, 명시적으로 선택해야)

### 4. 세 파일 작성

각 파일 템플릿은 아래 "파일 템플릿" 절 그대로 사용한다. 빈 헤더만 박지 말고
**대화에서 이미 합의된 내용으로 채워서 작성한다.** 합의되지 않은 항목은
"TBD — 첫 작업 착수 시 확정" 같은 명시적 플레이스홀더만 남긴다.

### 5. 사용자에게 보고

생성된 폴더 경로와 3개 파일 경로를 짧게 표시한다. 다음 단계(=실제 작업 착수)는
사용자가 별도 명령으로 지시할 때까지 시작하지 않는다. 이 스킬은 **기록 단계**이지
실행 단계가 아니다.

## 파일 템플릿

각 파일은 아래 골격을 따라 작성한다. 헤더 제목은 그대로, 본문은 대화 맥락으로 채운다.

### spec.md

```markdown
# spec — <slug>

## 목표
한 줄 정의. 이 작업이 완료됐을 때 무엇이 가능해지는가.

## 범위
- 포함:
  - …
- 제외:
  - …

## 완료 조건 (Definition of Done)
- [ ] …
- [ ] …

## 인터페이스 / 데이터 형식
필요한 경우 함수 시그니처, JSON 스키마, 엔드포인트, 파일 경로 등.

## 의존성
- 외부 라이브러리·서비스·CLI:
- 사전 작업으로 끝나야 하는 항목:

## 비고
설계상 의도적으로 빠뜨린 부분, 가정, 알려진 제약.
```

### context.md

```markdown
# context — <slug>

## 사용자의 원 요청
대화에서 사용자가 처음 이 작업을 꺼낸 말 (요약 또는 인용).

## 왜 이걸 지금 하는가
배경·동기. 어떤 문제·증상·요구가 이 작업을 트리거했는가.

## 결정된 방향
여러 대안 중 무엇을 선택했는가. 한 줄 요약.

## 기각된 대안
- 대안 A — 기각 이유
- 대안 B — 기각 이유

## 제약 / 합의 사항
- 기술적 제약:
- 시간·범위 제약:
- 사용자가 명시한 선호:

## 관련 자료
이 작업을 이해하는 데 필요한 파일·커밋·외부 문서 경로.
```

### checklist.md

```markdown
# checklist — <slug>

> 작업 진행하면서 AI 가 순차적으로 체크. `[x]` 로 표시한 항목은 완료된 것으로 간주.
> 새 항목이 발견되면 적절한 단계에 추가하고 체크리스트를 유지한다.

## 0. 준비
- [ ] spec.md / context.md 다시 한 번 읽고 어긋난 곳 없는지 확인
- [ ] 작업 브랜치 확인 또는 생성 (필요 시)

## 1. 구현
- [ ] …
- [ ] …

## 2. 검증
- [ ] 단위 동작 확인
- [ ] 회귀 확인 (영향 받는 기존 기능)
- [ ] (UI 가 있다면) 브라우저에서 직접 확인

## 3. 마무리
- [ ] 커밋 (CLAUDE.md commit_protocol 따름)
- [ ] checklist 의 미체크 항목이 남았으면 사유 메모
- [ ] 다음 세션이 픽업할 수 있도록 spec.md / context.md 의 변경분 반영
```

## 동작 원칙

- **파일은 한국어로 작성한다.** 사용자의 작업 언어가 한국어이므로 헤더만 위 영어 키워드를
  유지하고 본문 전체는 한국어로 쓴다. (slug 만 영문 kebab-case)
- **체크리스트는 살아있어야 한다.** 이 스킬이 끝난 뒤 사용자가 실제 작업을 시작하면,
  AI 는 작업 도중 `checklist.md` 의 항목을 `Edit` 으로 `[ ]` → `[x]` 갱신해야 한다.
  세 파일은 만든 뒤 잊지 말고 작업 흐름 중에 계속 참조한다.
- **한 번에 한두 개씩.** 작업 착수 후에는 체크리스트 항목을 통째로 몰아치지 않는다.
  1~2 개 단위로 진행하고 **매 단위마다** `checklist.md` 를 갱신한다. 그래야 중간에
  컨텍스트가 끊겨도 정확히 어디까지 했는지가 파일에 남는다. (10 개 파일을 동시에 고치고
  한참 뒤에 한꺼번에 보고하는 패턴 금지.)
- **세 파일은 자기 완결적으로.** "위에서 논의한 대로", "앞서 말한 방식" 같은 대화 의존
  표현을 파일에 쓰지 않는다. 이전 대화를 모르는 사람이 파일만 읽고도 전체 맥락을 복원할 수
  있어야 한다.
- **스펙 변경이 있으면 spec.md 부터 고치고 시작한다.** 코드와 스펙이 어긋난 채로 진행하면
  다음 세션이 거짓을 픽업한다.
- **context.md 는 거의 동결.** "왜 시작했는가" 는 사후에 바뀌지 않는다. 새로운 결정이
  생기면 context.md 끝에 `## 변경 이력` 섹션으로 append.
- **충돌하는 plan 폴더가 이미 있으면 절대 조용히 덮어쓰지 않는다.** 사용자 확인 필수.

## 새 대화에서 이어가기

이 스킬의 진짜 가치는 **다른 세션에서 이 폴더만 읽고 작업을 재개**할 수 있다는 데 있다.
사용자가 plan 폴더를 지목하며 이어가자고 하면 (예: "docs/plans/chat-panel-ai-bridge
이어서 해줘"), 다음 순서로 맥락을 복원한다.

1. `checklist.md` 읽기 → 지금 어디까지 했는지, 다음 미체크 항목이 무엇인지 파악
2. `context.md` 읽기 → 왜 이 작업을 하는지, 결정·제약·기각된 대안 확인
3. `spec.md` 읽기 → 전체 목표와 완료 조건 확인
4. 사용자에게 **"N/Total 완료 상태이고, 다음 작업은 X 입니다"** 형식으로 짧게 보고
5. 사용자 확인 후, 체크리스트의 남은 항목부터 이어서 진행

복원 보고 전에 코드를 먼저 손대지 않는다. 파일이 코드와 어긋나 있을 수 있으므로
"동작 원칙"의 *스펙 변경이 있으면 spec.md 부터* 규칙을 먼저 적용한다.

## 출력 예시

```
docs/plans/chat-panel-ai-bridge/
├── spec.md       (구현 스펙)
├── context.md    (시작 맥락)
└── checklist.md  (진행 체크리스트)

다음 세션에서 이 폴더만 읽으면 작업 재개 가능합니다.
```
